# Quay Dashboard v2 — Claude Instructions

## Project Overview
Static GitHub Pages dashboard (HTML + vanilla JS) for real-estate outbound calling performance.
Data driven by `quay/data.js`, which fetches `data/weekly_data.json` and `data/history.json` —
both auto-regenerated by GitHub Actions from the DialFire API (and, in future, ConnectTeams).
Live at https://twigs002.github.io/quay-dashboard-v2/.

Goal: operational visibility for the team + strategic overview for directors.

## Core Files & Architecture
- `index.html` — shell + script loading order (data.js → lib.js → views.js → app.js)
- `quay/styles.css` — design system (navy + brass-gold premium aesthetic, Montserrat + Permanent Marker)
- `quay/data.js` — REAL-DATA ADAPTER: fetches JSON, normalizes Dialfire fields,
  exposes `window.QUAY` with `agentsFor(period)`, `totalsFor`, `campaignsFor`,
  `DELTAS`, `WEEK_*`, `MONTH_*`, etc.
- `quay/lib.js` — `window.ICON` (SVG set) + `window.CHART` (weekly trend, donut, sparkline, miniBars)
- `quay/views.js` — secondary tab views (allStaff, compare, workTime, daily, manager, leadSources)
- `quay/app.js` — shell, sidebar, period state, router, Overview tab
- `data/weekly_data.json` — current snapshot (Dialfire-fed)
- `data/history.json` — 27 weeks of snapshots (Dialfire-fed)
- `scripts/fetch_dialfire.py` / `fetch_dialfire_daily.py` / `backfill_dialfire.py` — fetchers
- `.github/workflows/update-data.yml` / `update-daily.yml` / `backfill.yml` — schedule + manual triggers

## Data Model (per-agent record)
```
{ name, calls, success (=leads), seller, rental, email,
  cph, successRate, workTime, talkTime, wrapTime, pauseTime, waitTime,
  is_rm, meetsTarget, campaigns: [], talkPct, wrapPct, waitPct, workPct }
```
Plus `by_agent_campaign: { agent: { rawCampName: {calls, success, seller, rental, email, workTime, talkTime} } }`
which gives EXACT per-agent-per-campaign attribution (added 2026-06-05).
Historical weeks pre-dating this field fall back to overlap-counting in `campaignsFor`.

## Coding Standards
- Vanilla JS only — no React/Vue/heavy frameworks.
- Keep `data.js` the single source of truth for derived values; views consume, don't compute.
- Maintain mobile responsiveness; the sidebar auto-collapses below 1080px.
- camelCase JS; kebab-case CSS classes.
- One feature per commit where possible.
- Preserve the data pipeline — never break `data/*.json` consumers downstream.

## Script Load Order
`views.js` + `app.js` both capture `const Q = window.QUAY` at module-load time.
`index.html` defers loading them until `QUAY_READY` resolves. Don't break this.

## Agent Workflow Rules
1. Read current files before editing — never blind-rewrite.
2. Test changes by hard-refreshing https://twigs002.github.io/quay-dashboard-v2/
   (~30–60s GitHub Pages deploy lag after push).
3. Use Playwright smoke-tests for tab interactivity (see `/tmp/quay_dashv2_check.py` template).
4. Commit messages: `feat: …`, `fix: …`, `chore: …`. Group related edits.
5. Don't add the `Co-Authored-By` trailer (project policy).
